## Security

### 1. Plaintext Credential Storage

- **Severity**: Medium
- **Context**: API keys and OAuth tokens stored as plaintext in `~/.pi/agent/auth.json` and `~/.gemini/` files. No encryption at rest.
- **Mitigation**: This is consistent with pi's security model (user home directory permissions). No action required unless pi adds built-in keychain support.
- **Files**: `src/config-store.ts:41-48`, `src/oauth.ts`

### 2. API Key in Environment Variables

- **Severity**: Low
- **Context**: `GEMINI_API_KEY` / `GOOGLE_API_KEY` env vars can leak via `ps`, shell history, or CI logs.
- **Mitigation**: This is standard practice for CLI tools. Users should use `export` in their shell profile or a secrets manager.

### 3. No Token Scope Validation

- **Severity**: Low
- **Context**: OAuth tokens and API keys are accepted without verifying their scope/permissions.
- **Mitigation**: The Gemini API returns errors for insufficient permissions, which are surfaced through the error pipeline.

## Reliability

### 4. agy OAuth Token Cannot Be Refreshed

- **Severity**: Medium
- **Context**: agy's OAuth tokens expire after ~1 hour. The `refreshToken()` function is a no-op — it warns the user but returns the expired token anyway. Users on agy OAuth will see auth failures after ~1 hour.
- **Root cause**: agy CLI uses Google OAuth2 with a client ID we don't have access to. We can't call Google's token refresh endpoint on behalf of agy.
- **Workaround**: Users should switch to a static API key for long sessions.
- **Files**: `src/oauth.ts:82-95` (`refreshToken`)

### 5. Dynamic Model Discovery Failure Is Silent

- **Severity**: Low
- **Context**: If `fetchRemoteModels()` fails (network error, API change, etc.), it silently falls back to the static catalog. The user has no indication they're seeing stale models.
- **Mitigation**: This is by design — the extension must never crash pi on startup. Users who notice missing models should check `GEMINI_API_KEY`.
- **Files**: `src/models.ts:191-197` (`resolveModels`)

### 6. No Retry Logic for Transient Failures ✅ Resolved

- **Severity**: Low
- **Status**: Fixed — `fetchRemoteModels()` now retries once on transient network errors (with exponential backoff). HTTP error responses (4xx, 5xx) skip retries. Configurable via `retries`/`retryDelayMs` options.
- **Files**: `src/models.ts:130-179` (`fetchRemoteModels`)

## Potential Bugs

### 7. Race Condition: API Key Resolution vs. Model Discovery

- **Severity**: Low
- **Context**: In `src/index.ts`, `resolveApiKey()` is called before `resolveModels()`. If credentials change between calls (e.g., expired token), model discovery could fail while inference works, or vice versa.
- **Likelihood**: Very low (resolved from same sources).

### 8. String Trim for Token Files May Strip Valid Content

- **Severity**: Low
- **Context**: `walkAuthPaths()` trims raw file content: `parsed = raw.trim()`. If a token legitimately contains leading/trailing whitespace (unlikely but possible), it would be corrupted.
- **Likelihood**: Extremely low. OAuth tokens are alphanumeric.
- **Files**: `src/config-store.ts:75`

## Performance

### 13. Synchronous File Reads at Startup

- **Severity**: Low
- **Context**: `walkAuthPaths()` uses `readFileSync()` with the production default. Startup blocks on filesystem reads of credential files.
- **Impact**: Negligible — at most 3 small files read once at startup.
- **Files**: `src/config-store.ts:62`

### 14. No Model Discovery Caching

- **Severity**: Low
- **Context**: `fetchRemoteModels()` is called on every extension load (every pi session). Could cache results to disk with a 1-hour TTL.
- **Files**: `src/models.ts:130-179`
